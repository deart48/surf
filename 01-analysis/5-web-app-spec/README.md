> Этап 5. Технические задания (постановки на разработку) клиентского web-приложения
> кулинарной студии «Шеф-стол». В отличие от [этапа 3 (дизайн-бриф)](../3-design-brief/README.md),
> здесь описано **что и как реализует разработчик**: запросы к API, входные данные, элементы,
> состояния, логика, валидация и критерии приёмки.
>
> **Скоуп — клиентское web-приложение** (роль «Клиент»). Слоты, программы и шефы приходят из
> существующей инфраструктуры только для чтения. Интерфейсы шефа/владельца/админки — вне скоупа.

## Ключевые доменные правила (единые для всех ТЗ)

- **Вход без пароля:** телефон (E.164) + одноразовый код (OTP). Новый клиент дозаполняет имя (`PATCH /profile`).
- **Бронь:** `seats_count` 1..`min(free_seats, 6)` (сам клиент + до 5 гостей за одним столом);
  `rental_count` 0..`seats_count` и ≤ `free_rental_sets`. Прокат учитывается на стол отдельно от мест;
  своя экипировка занимает место, но не расходует прокатный фонд.
- **Цена:** `price_total = slot.price·seats_count + slot.rental_price·rental_count`. **Оплата офлайн** — онлайн-оплаты в UI нет.
- **Идемпотентность:** `createBooking` требует заголовок `Idempotency-Key`; повтор с тем же ключом и телом → тот же результат.
- **Отмена:** бесплатно **не позднее чем за 24 ч** (`cancelled`, места/прокат возвращаются); позже — `late_cancel`
  (не возвращаются, без штрафов). Ровно 24 ч — ранняя отмена.
- **Статусы:** бронь — `active`, `cancelled`, `late_cancel`, `studio_cancelled`; слот — `scheduled`, `cancelled`.
  «Прошедший» — производное от `slot.start_at`. История броней не скрывается.
- **Приватность:** клиент видит только свои профиль/брони; чужие данные → 403 (NFR-8).
