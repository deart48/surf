> **Скоуп — клиентское web-приложение и API для него.** Формирование расписания, интерфейсы
> шефа и владельца/администратора обеспечиваются существующей инфраструктурой и в этот список
> не входят (домен → «Границы скоупа», R-028).

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| BR-1 | Устранить ручную путаницу и двойные брони, возникающие при ведении записи через WhatsApp и Google-таблицу. Учёт записей должен вестись в единой системе, исключающей конфликты по местам за столом. | Must | Бриф (хаос в выходных, двойные записи за один стол); Ответ №1; Описание домена → «Процесс», «Цель клиентского приложения» |
| BR-2 | Дать клиентам возможность самостоятельно записываться на кулинарные классы онлайн без ручного посредничества владельца («люди сами записываются, а я только смотрю»). | Must | Бриф; Описание домена → «Цель клиентского приложения» |
| BR-3 | Снизить потери от неявок и поздних отмен (закупленные продукты пропадают) за счёт правила отмены 24 часов, фиксации поздних отмен и напоминаний — контроль без денежных штрафов («ограничить, но клиентов не злить»). | Must | Бриф (продукты закуплены и пропадают); Ответы №7, №8, №18; Описание домена → «Политика отмен» |
| BR-4 | На старте оплата остаётся офлайн (наличные / перевод на карту); продукт лишь фиксирует запись и показывает цену. Онлайн-оплата и отметка «оплачено/нет» — вне скоупа. | Must | Бриф; Ответы №11, №12, №13; Описание домена → «Оплата только офлайн» |
| BR-5 | Запустить продукт к началу нового сезона (жёсткий срок ~2 месяца) при ограниченном бюджете, начав с минимально необходимого набора функций (MVP-подход). | Must | Бриф (сроки и бюджет); Ответ №22; Описание домена → «Проект» |
| BR-6 | Клиентский интерфейс — web-приложение, потребляющее данные существующего бэкенда (слоты, программы, шефы) через API только для чтения. | Must | Бриф; Ответы №16, №21; Описание домена → «Технические / платформенные», R-028 |

## Метрики успеха

> Целевые значения — разумные ориентиры для MVP; при наличии данных уточняются.

| ID | Метрика | Целевой ориентир | Связь с требованиями |
| :-- | :-- | :-- | :-- |
| M-1 | Доля записей, сделанных клиентами онлайн самостоятельно (от общего числа записей за сезон). | ≥ 70% к концу первого сезона | BR-2 |
| M-2 | Число двойных броней / конфликтов по местам за столом. | 0 (полное устранение) | BR-1 |
| M-3 | Доля неявок и поздних отмен от числа записавшихся. | Снижение относительно текущего уровня | BR-3 |
| M-4 | Готовность к старту сезона. | Запуск в срок ~2 месяца с критичным набором функций (просмотр слотов, запись, отмены, аллергии) | BR-5 |

## Ограничения верхнего уровня

- **Срок и бюджет.** Жёсткая цель — начало нового сезона (~2 месяца); бюджет ограничен → строго MVP-подход, «навороты» откладываются. (Бриф; Ответ №22; домен → «Проект»)
- **Оплата офлайн на старте.** Онлайн-оплата и ручная отметка оплаты — вне границ скоупа. (Ответы №11–13; домен → «Оплата только офлайн»)
- **Web-first.** Клиентское приложение — веб (не нативное мобильное); данные получает из существующего бэкенда через API. (Ответ №21; домен → «Технические / платформенные»)
- **Граница серверной интеграции (R-004).** Транзакционность, SLA и маппинг сущностей — вне скоупа клиентского приложения; существующий бэкенд — black-box источник истины, гарантия «0 двойных броней» на его стороне. (домен → R-004)
- **Вне скоупа.** Интерфейсы шефа и владельца/администратора, создание/редактирование расписания, автоматический учёт форс-мажора — обеспечиваются существующей инфраструктурой. Онлайн-оплата и программа лояльности — направления развития. (домен → «Границы скоупа»)
- **Оценки шефов — Phase 2 (согласовано, R-001).** Оценка шефа звёздами осознанно вынесена из MVP в Phase 2, чтобы попасть в жёсткий срок начала сезона. (Бриф; Ответы №14, №15, №22; домен → «Границы скоупа», R-001)
