# Нефункциональные требования

> Этап 2. Как система должна работать: производительность, надёжность, безопасность,
> доступность, удобство. Частая ошибка — путать ФТ и НФТ.
> Источники: [Бриф](../0-customer-brief/brief-cooking.md),
> [Ответы заказчика](../1-elicitation/customer-questions.md),
> [Описание домена](../1-elicitation/domain-description.md).

> **Скоуп — клиентское web-приложение и API для него.** НФТ относятся к клиентскому приложению;
> функции шефа/владельца/админки обеспечиваются существующей инфраструктурой.

| ID | Категория | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- | :-- |
| NFR-1 | Платформа / Совместимость | Клиентский интерфейс — **web-приложение** (не нативное мобильное), работающее в современном браузере. | Высокий | Ответ №21; Описание домена → «Технические / платформенные» |
| NFR-2 | Удобство использования | Интерфейс рассчитан на самостоятельную запись без обучения: минимум полей и шагов, понятные состояния (в т.ч. empty state «Пока нет доступных классов»). | Высокий | Бриф («люди сами записываются»); R-027 |
| NFR-3 | Удобство использования | Лёгкая регистрация/вход: только имя и телефон с подтверждением по коду (OTP), без сложных паролей. Порог входа для клиента минимален. | Высокий | Ответ №6 |
| NFR-4 | Надёжность / Целостность данных | При одновременной записи нескольких клиентов на один слот исключаются двойные брони и овербукинг: число занятых мест и прокатных комплектов никогда не превышает лимитов слота/стола. Это критическое требование (главная боль заказчика); гарантия «0 двойных броней» обеспечивается на стороне бэкенда (black-box), клиент полагается на ответы API. | Высокий (Must) | Бриф (двойные записи); Ответы №1, №2; Описание домена → R-004 |
| NFR-5 | Надёжность / Целостность данных | Состояние записей, свободных мест и прокатного фонда остаётся согласованным при отменах и параллельных операциях (ранняя отмена корректно освобождает места и прокатные комплекты, без потери и задвоения данных). | Высокий | Описание домена → «Политика отмен», «Процесс» |
| NFR-6 | Производительность | Список слотов и подтверждение записи откликаются быстро при нормальной сети; система сохраняет приемлемую отзывчивость в пиковые часы выходных, когда одновременно записывается наибольшее число клиентов. | Высокий | Бриф (в выходные — «ад»); Ответ №22 |
| NFR-7 | Безопасность / Приватность | Персональные данные клиента (имя, телефон) защищены от несанкционированного доступа; хранятся и передаются безопасно. | Высокий | Ответ №6; Описание домена → «Пользователи и роли» |
| NFR-8 | Безопасность / Разграничение доступа | Клиент видит и управляет только своими бронированиями и профилем; доступ к чужим бронированиям и к данным управления (расписание, программы, шефы на запись) через клиентский API запрещён. Слоты, программы и шефы доступны клиенту только для чтения. | Высокий | Ответы №4, №16; Описание домена → «Границы скоупа», R-004 |
| NFR-9 | Уведомления | Push-напоминание о записи доставляется заблаговременно — **за 24 часа** до старта класса, чтобы снизить неявки. При отсутствии разрешения на уведомления приложение работает штатно без push. | Средний | R-006; Ответ №18; Описание домена → «Границы скоупа» |
| NFR-10 | Интеграция с бэкендом | Клиентское приложение взаимодействует с существующим бэкендом через API; каноническая схема данных — контракт API (`01-analysis/api/`). Приложение не создаёт и не редактирует слоты, программы и шефов; корректно обрабатывает отказ бэкенда (например, мест уже нет). | Высокий | R-004, R-015; Ответ №5; Описание домена → «Технические / платформенные» |
| NFR-11 | Ограничения проекта | Решение должно быть запущено к началу нового сезона — жёсткий срок ~2 месяца — при ограниченном бюджете; применяется MVP-подход, «навороты» (онлайн-оплата, лояльность, оценки) откладываются. | Высокий (Must) | Бриф; Ответы №20, №22; Описание домена → «Проект» |
