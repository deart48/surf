Приоритеты по MoSCoW: **Must** — обязательно к запуску; **Should** — желательно при наличии времени; **Could** — приятное дополнение; **Won't** — вне границ скоупа.

## Границы скоупа (клиентское приложение + API)

Скоуп — **клиентское web-приложение и API для него**. Функции шефа, владельца и администратора
обеспечиваются **существующей инфраструктурой** и здесь не описываются.

Следствия для скоупа:

- **Слоты (классы) приходят из существующего бэкенда через API.** Экранов создания и редактирования расписания в клиентском приложении нет.
- **Программы/меню и шефы** доступны клиенту как справочные (read-only) данные из API — клиент видит их в карточке слота, но не управляет ими.

## Запись на класс (клиент)

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| FR-6 | Система должна позволять клиенту записаться на выбранный слот в пределах свободных мест. | Must | Бриф; Описание домена → «Процесс» |
| FR-7 | Система должна позволять клиенту при записи для каждого места выбрать вариант экипировки (фартук + набор ножей): своя или прокатная. | Must | Бриф; Ответ №2; Описание домена → «Экипировка» |
| FR-8 | Система должна позволять клиенту одной записью забронировать несколько мест — **не более 6 человек на одну бронь (себя + до 5 гостей), в пределах одного стола** — при наличии свободных мест. По гостям приложение собирает только **количество мест** и **выбор экипировки** (своя/прокат); имена и контакты гостей не собираются, ответственность за гостей несёт клиент, оформивший бронь. | Must | Ответы №3, №10; Описание домена → «Бизнес-ограничения» (одна бронь — максимум 6 человек) |
| FR-9 | Система должна ограничивать число доступных к бронированию мест слота величиной `min(свободные места слота, 6)`, где 6 — лимит мест на одну бронь (клиент + до 5 гостей, в пределах одного стола). Число мест слота приходит из существующей инфраструктуры. | Must | Ответы №1, №3; Описание домена → «Бизнес-ограничения» (12 столов × до 6 человек; вместимость в людях) |
| FR-10 | Система должна вести **отдельный** учёт прокатного фонда независимо от учёта мест: число прокатных комплектов в брони ограничено свободным прокатным фондом на стол (до 6 комплектов на стол). Вариант «своя экипировка» занимает место, но **не** занимает прокатный комплект из фонда. | Must | Ответ №2; Описание домена → «Бизнес-ограничения», «Своя экипировка» |
| FR-11 | Система должна запрещать запись, превышающую лимит мест (FR-9) или лимит свободных прокатных комплектов (FR-10), исключая двойную бронь и овербукинг. При отказе бэкенда (мест уже нет) приложение корректно обрабатывает ошибку и показывает актуальные данные. | Must | Бриф (двойные записи); Ответ №2; Описание домена → R-004 |
| FR-12 | Система должна позволять клиенту указать аллергии **одним полем на всю бронь** (опционально) и передать их шефу через бэкенд. Сбор аллергий по каждому участнику отдельно не выполняется. | Must | Ответ №9; Описание домена → «Бизнес-ограничения» (Аллергии), «Экипировка» |
| FR-13 | Система должна показывать клиенту цену класса и фиксировать запись; оплата производится офлайн (наличные / перевод). Отметки «оплачено/нет» и онлайн-оплаты нет. | Must | Ответы №11, №12, №13; Описание домена → «Оплата только офлайн» |

## Мои бронирования и отмены (клиент)

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| FR-14 | Система должна показывать клиенту список его бронирований со статусом, параметрами слота, числом мест и вариантом экипировки. В списке отображаются в том числе отменённые брони и поздние отмены. | Must | Описание домена → «Цель клиентского приложения» (свои бронирования); Ответ №8 |
| FR-15 | Система должна позволять клиенту отменить свою запись до старта класса. Отмена возможна только **целиком**; частичная отмена отдельного места или гостя не поддерживается. | Must | Бриф; Описание домена → «Процесс», «Отмена» |
| FR-16 | Система должна при ранней отмене (**не позднее чем за 24 часа** до старта; ровно за 24 ч включительно считается ещё ранней) освобождать забронированные места обратно в слот и возвращать прокатные комплекты в фонд. | Must | Ответ №7; Описание домена → «Политика отмен» |
| FR-17 | Система должна при поздней отмене (**менее чем за 24 часа** до старта) фиксировать запись статусом «поздняя отмена» без освобождения места и без денежных штрафов. | Must | Ответ №8; Описание домена → «Политика отмен» |
| FR-18 | Система должна отображать брони, отменённые студией (организатором / по форс-мажору): бронь не исчезает из истории, а переходит в статус «Отменён студией» с указанием причины; клиенту приходит push-уведомление; повторная запись на отменённый студией слот запрещена. Само действие отмены слота выполняется в существующей инфраструктуре. | Must | R-008; Ответ №19; Описание домена → «Отмена класса студией / по форс-мажору» |
