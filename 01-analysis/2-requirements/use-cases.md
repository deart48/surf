# Use cases / сценарии

> Этап 2. Сценарии использования. Обязательно прорабатываем **альтернативные потоки** и
> **исключения** — их легко забыть.
>
> **Скоуп: клиентское web-приложение и API для него.** Слоты, программы и шефы поступают из
> существующего бэкенда через API (read-only).
> Источники: [Ответы заказчика](../1-elicitation/customer-questions.md),
> [Описание домена](../1-elicitation/domain-description.md).

## UC-1: Просмотр и фильтрация списка классов

- **Актёры:** Клиент
- **Приоритет:** Must
- **Предусловия:** Клиент авторизован.
- **Триггер:** Клиент хочет найти подходящий кулинарный класс.
- **Источник:** R-027; Ответ №5; Описание домена → «Период списка слотов по умолчанию».

### Основной поток
1. Клиент открывает список классов. По умолчанию показываются слоты на **ближайшие 7 дней** от текущего момента, отсортированные по времени старта.
2. Каждый слот показывает дату/время, программу, шефа, цену, всего и свободно мест.
3. (Опц.) Клиент задаёт фильтры: дата/период, тип программы (новичковый/опытный), наличие свободных мест, шеф.
4. Система показывает отфильтрованный список с ключевыми параметрами каждого слота.

### Альтернативные потоки
- **A1. Больший период.** Клиент задаёт диапазон дат шире 7 дней через фильтр дат → показываются слоты за выбранный период.
- **A2. Сброс фильтров.** Клиент очищает фильтры → возвращается дефолтная выдача (ближайшие 7 дней).

### Исключения / ошибки
- **E1. Нет классов под условия.** Фильтр не дал результатов → пустое состояние с подсказкой изменить/сбросить фильтры.
- **E2. Нет расписания вовсе.** В горизонте выдачи нет ни одного предстоящего слота → empty state «Пока нет доступных классов».

### Постусловия
- Клиент видит список предстоящих классов; данные расписания не изменяются.

## UC-2: Запись на класс (себя и гостей)

- **Актёры:** Клиент
- **Приоритет:** Must
- **Предусловия:** Клиент авторизован; есть слот со свободными местами.
- **Триггер:** Клиент хочет забронировать место на классе.
- **Источник:** Ответы №1, №2, №3, №9, №10; Описание домена → «Процесс», «Бизнес-ограничения».

> **Состав гостей.** По гостям собираются только **количество мест** и **выбор экипировки** (своя/прокат). Имена и контакты гостей не собираются; ответственность за гостей несёт клиент, оформивший бронь (Ответ №10).
>
> **Граница серверной интеграции (R-004).** Атомарная проверка свободных мест и гарантия «0 двойных броней» — на стороне бэкенда. Клиент полагается на ответы API и корректно обрабатывает отказ.

### Основной поток
1. Клиент открывает карточку слота и просматривает параметры (дата/время, программа, шеф, цена, адрес, свободно мест, доступность прокатных комплектов).
2. Указывает число мест (себя + до 5 гостей, всего до 6, в пределах одного стола).
3. Для каждого места выбирает вариант экипировки: своя / прокатная.
4. (Опц.) Указывает аллергии — одним полем на всю бронь.
5. Подтверждает запись.
6. Система проверяет лимит мест `min(свободные места, 6)` и **отдельно** прокатный фонд стола (до 6 комплектов на стол), создаёт запись, уменьшает счётчики свободных мест и прокатных комплектов.
7. Система показывает подтверждение с ценой (оплата офлайн) и добавляет запись в «Мои бронирования».

### Альтернативные потоки
- **A1. Вся экипировка — своя.** На шаге 3 прокатный фонд не затрагивается; уменьшается только число мест.
- **A2. Часть мест — прокат.** На шаге 6 прокатный фонд уменьшается только на число «прокатных» мест.

### Исключения / ошибки
- **E1. Не хватает мест.** Запрошено больше, чем свободно мест → запись отклоняется, показывается актуальное число свободных мест.
- **E2. Не хватает прокатных комплектов.** Прокатных мест запрошено больше, чем свободно комплектов на стол (места при этом могут быть) → система предлагает уменьшить число прокатных мест или выбрать «своя экипировка».
- **E3. Гонка запросов / слот заполнился параллельно.** К моменту подтверждения места заняты другим клиентом → запись отклоняется без овербукинга и двойной брони, список обновляется.
- **E4. Слот отменён студией.** К моменту подтверждения слот отменён студией → запись недоступна; повторная запись на отменённый слот запрещена (R-008).

### Постусловия
- При успехе: создана запись клиента; счётчики слота и прокатного фонда уменьшены; запись видна в «Мои бронирования».
- При ошибке: состояние слота не изменилось.

## UC-3: Отмена записи

- **Актёры:** Клиент
- **Приоритет:** Must
- **Предусловия:** У клиента есть предстоящая запись; класс ещё не стартовал.
- **Триггер:** Клиент решает отменить участие.
- **Источник:** Ответы №7, №8; Описание домена → «Политика отмен».

> **Объём отмены.** Отмена выполняется **только целиком**: отменяется вся бронь со всеми местами и прокатными комплектами. Частичная отмена (например, одного гостя) не поддерживается.
>
> **Граница политики.** Ровно за 24 часа до старта включительно — ещё **ранняя** (бесплатная) отмена; менее 24 часов — **поздняя**. Источник истины по времени и статусу — бэкенд.

### Основной поток
1. Клиент открывает «Мои бронирования» и выбирает предстоящую запись.
2. Нажимает «Отменить» (отмена всей брони целиком).
3. Сервер определяет время до старта.
4. **Ранняя отмена (≥24 ч):** вся бронь помечается «отменена»; все места и прокатные комплекты возвращаются в слот/фонд.
5. Система показывает подтверждение отмены.

### Альтернативные потоки
- **A1. Поздняя отмена (<24 ч).** На шаге 4 запись помечается «поздняя отмена»; места и прокатные комплекты НЕ освобождаются; денежных штрафов нет; клиенту показывается пояснение политики 24 часов.

### Исключения / ошибки
- **E1. Класс уже стартовал/прошёл.** Кнопка отмены недоступна; запись нельзя отменить.
- **E2. Запись уже отменена.** Повторная отмена не выполняется; показывается текущий статус.

### Постусловия
- Статус всей брони обновлён («отменена» / «поздняя отмена»).
- При ранней отмене все места и прокатные комплекты брони снова доступны другим клиентам.

## UC-4: Отмена класса студией (форс-мажор)

- **Актёры:** Клиент (получатель), существующая инфраструктура (инициатор отмены — вне скоупа)
- **Приоритет:** Must
- **Предусловия:** У клиента есть бронь на слот, который студия отменяет (срыв поставки продуктов / поломка оборудования).
- **Триггер:** Студия отменяет класс в существующей инфраструктуре.
- **Источник:** R-008; Ответ №19; Описание домена → «Отмена класса студией / по форс-мажору».

### Основной поток
1. Студия отменяет слот в существующей инфраструктуре (вне скоупа клиентского приложения).
2. Бэкенд переводит бронь клиента в статус **«Отменён студией»** с указанием причины (бронь не удаляется).
3. Клиенту отправляется push-уведомление об отмене.
4. В списке и деталях брони отображается статус «Отменён студией» и причина; кнопка записи на этот слот неактивна.

### Альтернативные потоки
- **A1. Нет разрешения на push.** Уведомление не доставляется, но статус «Отменён студией» и причина видны в «Моих бронированиях» при открытии приложения.

### Исключения / ошибки
- **E1. Попытка повторной записи на отменённый слот.** Запись отклоняется — повторная запись на отменённый студией слот запрещена.

### Постусловия
- Бронь клиента сохранена со статусом «Отменён студией» и причиной; повторная запись на слот недоступна.

## UC-5: Вход по номеру телефона (OTP)

- **Актёры:** Клиент
- **Приоритет:** Must
- **Предусловия:** У клиента есть доступ к номеру телефона.
- **Триггер:** Клиент хочет войти в приложение.
- **Источник:** Ответ №6; Описание домена → «Границы скоупа» (вход по телефону/OTP).

### Основной поток
1. Клиент вводит имя (при первом входе) и номер телефона.
2. Система отправляет одноразовый код (OTP) на номер.
3. Клиент вводит код.
4. Система подтверждает личность и предоставляет доступ.

### Альтернативные потоки
- **A1. Повторная отправка кода.** Клиент запрашивает код заново, если не получил предыдущий.

### Исключения / ошибки
- **E1. Неверный код.** Показывается сообщение «неверный код»; поле ввода остаётся активным для повторной попытки.

### Постусловия
- Клиент авторизован и может просматривать классы, записываться и управлять своими бронированиями.
