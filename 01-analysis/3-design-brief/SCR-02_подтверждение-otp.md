# SCR-02 · Подтверждение кода (OTP)

- **Роль:** Клиент (единственная роль в скоупе)
- **Приоритет:** Must
- **Связанные требования:** FR-2, UC-5 (шаги 3–4, A1 повторная отправка, E1 неверный код), US-1
- **Точки входа:** с **SCR-01 «Вход: телефон и имя»** сразу после успешной отправки кода (передаётся номер телефона).
- **Точки выхода:** успешное подтверждение → раздел **«Классы» (SCR-03)**; «Изменить номер» / назад → возврат на SCR-01.

---

## 1. Роль экрана и задача пользователя

Второй и последний шаг лёгкого входа. Клиент только что попросил код — теперь ему нужно ввести короткий одноразовый код из СМС и попасть в приложение к классам. Задача пользователя предельно узкая, поэтому экран должен быть максимально сфокусированным: ввод кода, подтверждение, и всё. Любое трение здесь — прямая потеря клиента у самой двери.

Экран также страхует типичные заминки: «код не пришёл» (повторная отправка по таймеру) и «ввёл не тот код» (понятная ошибка без сброса всего процесса).

## 2. Контекст, точки входа и выхода

Клиент приходит сюда только с SCR-01, где ввёл номер и нажал «Получить код». Номер передаётся на этот экран и **показывается**, чтобы человек убедился, что код ушёл именно туда, куда он ожидал. Отсюда два выхода: успех ведёт в «Классы» (SCR-03) — сердце приложения, ради которого всё затевалось; а «Изменить номер» возвращает на SCR-01, если клиент опечатался в телефоне.

Экран живёт в связке с SCR-01 как единый мини-флоу входа. Никаких паролей, альтернативных каналов (email, соцсети) и «запасных вопросов» — только OTP.

## 3. Информационная архитектура и контент (приоритеты)

Сверху вниз по важности:

1. **Заголовок и пояснение** — «Введите код из СМС», под ним номер, на который отправлен код (например «Код отправлен на +7 900 123-45-67»).
2. **Поле ввода кода** — центральный элемент, крупный, удобный для цифр.
3. **Кнопка «Подтвердить»** — основное действие.
4. **Блок повторной отправки** — таймер обратного отсчёта и появляющаяся по его истечении ссылка/кнопка «Отправить код повторно» (A1).
5. **«Изменить номер» / назад** — путь исправить телефон, если он введён с ошибкой.
6. **Область сообщений об ошибке** — «неверный код», «код истёк», «слишком много попыток».

Никакого лишнего контента: экран должен читаться как одно короткое действие.

## 4. Ключевые блоки и элементы интерфейса

- **Поле ввода кода.** Удобный ввод цифр — раздельные ячейки под каждый символ кода (наиболее ожидаемый паттерн OTP) либо одно поле; дизайнер выбирает, но с учётом требований: автопереход между ячейками по мере набора, автоматическая обработка **вставки всего кода из буфера** (код распределяется по ячейкам целиком), удаление с возвратом фокуса назад. Тип ввода — числовой (цифровая клавиатура на mobile web).
- **Показ номера получателя.** Строка с номером, на который ушёл код, чтобы клиент видел адресата и мог заметить опечатку.
- **Кнопка «Подтвердить».** Первичная. Неактивна, пока код не введён полностью; в состоянии проверки — с индикатором загрузки и заблокирована. По требованию дизайнер решает, подтверждать ли автоматически при вводе последней цифры, оставляя кнопку как явный дубль действия.
- **Таймер и «Отправить код повторно» (A1).** До истечения таймера показываем отсчёт («Запросить новый код можно через 0:45»); по нулю таймер сменяется активной ссылкой «Отправить код повторно». После повторной отправки таймер запускается заново.
- **«Изменить номер».** Заметная, но вторичная ссылка/кнопка возврата на SCR-01 с предзаполненным номером.
- **Область ошибок.** Место под полем для текста ошибки без скачков макета.

## 5. Сценарии и взаимодействия

- **Основной (UC-5, шаги 3–4).** Клиент вводит код → нажимает «Подтвердить» (или срабатывает автоподтверждение) → кнопка/поле в состоянии проверки → при успехе переход в «Классы» (SCR-03).
- **A1. Повторная отправка кода.** Клиент не получил СМС → ждёт окончания таймера → жмёт «Отправить код повторно» → приходит новый код, поле очищается/готово к новому вводу, таймер стартует заново. Прежний код может стать недействительным — это нормально.
- **E1. Неверный код.** Введён неправильный код → сообщение «Неверный код, попробуйте ещё раз»; **поле остаётся активным** (не блокируется и не сбрасывает флоу), ошибочный ввод подсвечивается и очищается для новой попытки, фокус — в начало поля.
- **Истёкший код.** Если код устарел (по времени) — сообщение «Срок действия кода истёк» и явное приглашение запросить новый (через блок повторной отправки).
- **Изменить номер.** Клиент понимает, что ошибся в телефоне → возврат на SCR-01, где номер предзаполнен для правки.
- **Вставка из буфера / автозаполнение.** Код, скопированный из СМС, вставляется целиком и раскладывается по ячейкам; на поддерживающих браузерах используется автозаполнение одноразового кода (см. раздел 9).
- **Enter.** При полном коде Enter эквивалентен «Подтвердить».

## 6. Состояния экрана

- **Ожидание ввода (idle).** Поле пустое и в фокусе, номер получателя показан, таймер повторной отправки идёт, кнопка «Подтвердить» неактивна.
- **Проверка (loading).** После отправки кода — индикатор на кнопке/поле, ввод заблокирован от повторной отправки, пока идёт проверка.
- **Ошибка «неверный код» (E1).** Понятное сообщение, поле активно для повторной попытки, флоу не сбрасывается, счётчик оставшихся попыток при необходимости отражён.
- **Истёкший код.** Сообщение об истечении + приглашение запросить новый; ввод старого кода больше не приводит к успеху.
- **Лимит попыток / повторных отправок.** Если бэкенд вернул превышение лимита (много неверных вводов или частые запросы кода) — состояние временной блокировки с понятным объяснением и, по возможности, когда можно повторить; действия временно неактивны.
- **Успех.** Кратко подтверждаем и уводим в «Классы» (SCR-03) — без лишней задержки и без промежуточных экранов.
- **Нет доступа.** Прямой заход на SCR-02 без пройденного SCR-01 (нет номера/контекста) — экран не имеет смысла: возвращаем на SCR-01.

## 7. Крайние случаи, ограничения и валидация

- **Формат кода.** Клиент проверяет только длину/цифровой состав; истинность кода определяет бэкенд (NFR-10) — UI не «знает» правильный код.
- **Поле остаётся активным при ошибке (E1)** — критично: неверный код не должен выбрасывать клиента назад или блокировать поле.
- **Лимиты — на стороне бэкенда.** Число попыток ввода и частота повторных отправок ограничиваются сервером; UI лишь корректно отображает отказ и, если сервер сообщает время, показывает, когда можно повторить. Точные числа лимитов и длину кода UI не хардкодит.
- **Таймер повторной отправки** ведёт себя предсказуемо: не даёт спамить запросами кода, перезапускается после каждой отправки; при возврате на экран отсчёт должен быть консистентным.
- **Смена номера сбрасывает контекст** — при возврате на SCR-01 и новом номере начинается новый цикл OTP.
- **Сбой сети при подтверждении/повторной отправке** — понятное сообщение и возможность повторить без потери введённого кода, где это возможно.
- **Автозаполнение из СМС** может подставить код автоматически — дизайн должен корректно отработать как ручной ввод, так и автоподстановку (в т.ч. автоподтверждение, если оно предусмотрено).

## 8. Тон, иерархия и визуальные акценты

Спокойно, уверенно, без суеты — клиент в одном шаге от классов. Главный акцент — поле кода и «Подтвердить». Номер получателя — вторичный, но заметный ориентир доверия. Блок повторной отправки не должен провоцировать тревогу («код не пришёл?!») — подаётся мягко, как страховка. Ошибки формулируются по-человечески и подсказывают следующий шаг («Неверный код, попробуйте ещё раз», «Код устарел — запросите новый»), в духе тёплого тона студии, без канцелярита и обвинений.

## 9. Доступность и web-специфика

- **Адаптивность.** Desktop web — компактная центрированная карточка; ячейки кода не растягиваются неестественно. Mobile web — ячейки крупные, удобные для пальца, не перекрываются экранной клавиатурой.
- **Клавиатура и фокус.** При открытии фокус сразу в поле кода. Внутри раздельных ячеек — автопереход вперёд при вводе и назад при удалении, видимый фокус на активной ячейке. Enter подтверждает при полном коде. Порядок Tab: поле кода → «Подтвердить» → повторная отправка → «Изменить номер».
- **Мобильная клавиатура.** Числовой режим ввода для кода.
- **Автозаполнение одноразового кода.** Использовать web-семантику автозаполнения OTP (`autocomplete="one-time-code"`), чтобы браузер/ОС предложили код из СМС; поддержать вставку кода целиком из буфера.
- **Доступность.** Поле кода имеет доступную подпись; сообщения об ошибке и статус («идёт проверка», «неверный код») озвучиваются скринридером; таймер повторной отправки доступен не только цветом/анимацией; достаточный контраст текста, ячеек и кнопок.

## 10. Критерии готовности дизайна (чек-лист)

- [ ] Показан номер, на который отправлен код, и путь «Изменить номер» (возврат на SCR-01).
- [ ] Поле кода с раздельными ячейками (или эквивалент): автопереход, удаление с возвратом, вставка кода целиком.
- [ ] Кнопка «Подтвердить» с состояниями: неактивна / активна / проверка (заблокирована); поведение автоподтверждения определено.
- [ ] Таймер обратного отсчёта и «Отправить код повторно» (A1) с перезапуском таймера.
- [ ] Отрисованы состояния: ожидание ввода, проверка, неверный код (поле активно, E1), истёкший код, лимит попыток/повторных отправок, успех.
- [ ] Обработан прямой заход без контекста номера (возврат на SCR-01).
- [ ] Успешное подтверждение ведёт в «Классы» (SCR-03).
- [ ] Адаптив desktop/mobile web; числовая клавиатура; автозаполнение OTP из СМС; видимый фокус; Enter подтверждает.

## 11. Открытые вопросы на согласование

- **Автоподтверждение.** Подтверждать ли вход автоматически по вводу последней цифры (и оставлять ли кнопку «Подтвердить» как явный дубль)?
- **Длина кода и параметры таймера.** Сколько цифр в коде, длительность обратного отсчёта до повторной отправки и допустимое число повторных отправок? Значения приходят с бэкенда — нужны для точной вёрстки ячеек и текстов.
- **Лимиты попыток.** Как именно показывать блокировку при превышении числа неверных вводов (текст, время до разблокировки)?
- **Возврат после блокировки.** Что видит клиент, если превышены и попытки, и лимит повторных отправок — предлагать ли только «Изменить номер» / попробовать позже?
