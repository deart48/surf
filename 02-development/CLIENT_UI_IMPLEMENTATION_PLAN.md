# План реализации UI клиентского web-приложения «Шеф-стол» (без бэкенда)

> Итеративный чек-лист для реализации экранов `SCR-01…SCR-10` на статичных/mock-данных,
> без подключения к реальному API. Цель — визуально и функционально (в рамках клиентского
> состояния) проверить экраны из ТЗ ([`01-analysis/5-web-app-spec/`](../01-analysis/5-web-app-spec/README.md))
> и доменные правила, прежде чем подключать NestJS backend.
>
> **Дизайн-референс:** [Restaurant UI Web Kit (Figma Community)](https://www.figma.com/design/jMX4FTxPRWbGSPK0Do7C7z/Restaurant-UI-Web-Kit--Community-)
> используется только как источник визуального языка (тёплая палитра, карточки, типографика
> кулинарной тематики). **Контент, поля, состояния и переходы должны соответствовать ТЗ
> (`SCR-*`), а не структуре макета** — макет про ресторан/меню, ТЗ — про запись на кулинарные
> классы «Шеф-стол». Где макет и ТЗ расходятся, побеждает ТЗ.

## Как читать этот план

- Каждый пункт — отдельная задача. Отмечайте `[x]` по мере выполнения.
- Пункты сгруппированы по вертикальным срезам (каркас → экран → экран...).
- В разделе [Отчёт о реализации](#отчёт-о-реализации) в конце фиксируется, что сделано,
  что не удалось и почему — заполняется по итогам работы.

## Отчёт о реализации

> этапа (подключение NestJS backend по `BE_IMPLEMENTATION_PLAN.md`).

### Реализовано

- Каркас: Vite + React 19 + TypeScript + React Router 7 + Tailwind CSS v4 в `web/`; `npm run
  build`, `npx tsc -b` и `npm run lint` (oxlint) проходят чисто (1 безобидный `react-refresh`
  warning в файле контекста сессии).
- Полный mock-слой (`src/mock/`) без единого реального HTTP-запроса к backend: фикстуры
  программ/шефов/слотов/клиента/броней и функции с сигнатурами `operationId` из
  `01-analysis/api` (`listSlots`, `getSlot`, `listPrograms`, `listChefs`, `requestAuthCode`,
  `verifyAuthCode`, `updateProfile`, `createBooking`, `listBookings`, `getBooking`,
  `cancelBooking`). Состояние живёт в `localStorage`, так что брони/отмены переживают
  перезагрузку страницы в рамках сессии.
- Доменные политики вынесены в чистый модуль `src/domain/policies.ts`: лимиты мест/проката,
  расчёт цены, правило ранней/поздней отмены (24 ч), форматирование даты/денег, валидация
  E.164 — без зависимостей от React/API, готовы к переиспользованию при подключении backend.
- Все 10 экранов из ТЗ реализованы и связаны роутингом с route guard по mock-сессии:
  - SCR-01/SCR-02 — вход по телефону + OTP (4 цифры, paste, таймер повторной отправки, форма
    имени для нового клиента, ошибка неверного кода).
  - SCR-03/SCR-04 — список классов за 7 дней, карточки, фильтры по типу программы/шефу/
    доступности, «показать ещё», empty state.
  - SCR-05 — карточка класса со всеми параметрами слота и корректными disabled-состояниями CTA.
  - SCR-06/SCR-07 — форма записи с live-пересчётом цены, лимитами мест/проката, полем аллергий,
    защитой от двойного сабмита, обработкой mock-ошибок 409/410; экран успеха с mock push-опцией.
  - SCR-08/SCR-09 — мои бронирования (вкладки «Предстоящие»/«История»), детали брони, отмена с
    модальным предупреждением о раннем/позднем сценарии.
  - SCR-10 — профиль: имя/телефон, редактирование имени, переключатель push, выход с
    подтверждением.
- Сквозные состояния Loading/Empty/Error как переиспользуемые компоненты (`Spinner`,
  `EmptyState`, `ErrorState`) применены на всех списках и карточках.
- Доступность: фокус-ловушка + `Esc` + возврат фокуса в модалках (`shared/ui.tsx#Modal`),
  видимый `:focus-visible`, `aria-live`/`role="alert"` на ошибках и OTP-полях.
- Дизайн — тёплая палитра (терракота/олива/крем), карточки, скруглённые CTA, sticky-кнопки на
  мобильной ширине — по мотивам визуального языка Figma Restaurant UI Kit, но контент и структура
  экранов взяты строго из ТЗ (см. отклонения ниже).

### Не реализовано / отложено

- **Реальная визуальная проверка в браузере.** Sandbox-окружение агента изолирует сеть между
  вызовами инструментов: фоновой `vite dev` процесс запускается в одном сетевом namespace, а
  последующие `curl`/browser-инструмент — в другом, поэтому `localhost` недоступен извне того же
  вызова, где сервер был запущён. Проверка выполнена статически: `tsc -b` (типы), `vite build`
  (сборка), `oxlint` (линт) — все чистые, но ручного клика по экранам в реальном окне браузера не
  было. Рекомендуется открыть `npm run dev` локально и пройтись по чек-листу вручную.
- **Error Boundary** (React) — не добавлен; сейчас необработанное исключение в рендере уронит всё
  дерево. Для mock-этапа риск невысокий (данные строго типизированы), но стоит добавить перед
  реальной интеграцией.
- **Фильтр периода дат** (`date_from`/`date_to`) на SCR-04 — не реализован; сейчас список всегда
  использует дефолтные 7 дней от текущего момента. Раздельные UI-варианты «bottom sheet на mobile
  / боковая панель на desktop» тоже не сделаны — используется единая модалка на всех ширинах.
- Явный счётчик «найдено N классов» рядом с кнопкой фильтров — не выведен отдельным элементом.
- Web Push — только `Notification.requestPermission()` best-effort; полноценная регистрация
  Service Worker/Push Subscription не входит в mock-этап и требует backend для отправки.
- i18n/мультиязычность, тёмная тема, полноценное покрытие тестами (unit/e2e) — не запрашивались
  и не делались.

### Сознательные отклонения от Figma-референса в пользу ТЗ

- Макет Restaurant UI Web Kit — про ресторан/меню/заказ блюд; предметная область ТЗ — запись на
  кулинарные классы («Шеф-стол»). Экранная структура, поля форм, статусы и переходы взяты **только
  из `SCR-*`/`LOGIC-*`**, макет использован исключительно как источник тона (тёплая
  терракотово-оливковая палитра, крупная типографика заголовков, карточки со скруглением,
  капсульные кнопки) — никаких карточек «блюдо + добавить в корзину», меню ресторана, категорий
  кухни и т.п. из макета в проект не переносилось.
- Счётчик проката сделан агрегированным (`rental_count: 0..seats_count`), а не как переключатель
  «своё/прокат» у каждого места по отдельности, — так буквально устроена модель данных API
  (`seats_count` + `rental_count`, не массив мест), это соответствует ТЗ точнее, чем более похожая
  на макет идея «карточка на каждого гостя».
- Имя нового клиента запрашивается отдельным шагом после успешного OTP, а не на первом экране —
  так пользователь вводит телефон один раз и не отвлекается на лишнее поле, если он уже
  зарегистрирован; это соответствует духу UC-1 (минимум шагов для повторного входа).

[«10. Сетевая интеграция с backend»](#10-сетевая-интеграция-с-backend-после-mock-этапа) выше).
Актуальные рекомендации на следующий шаг:

1. **Поднять живую PostgreSQL и прогнать полный ручной сценарий** (`backend/NEXT_STEPS.md` п.1–2)
   — ни один HTTP-запрос между web и backend ни разу не выполнялся по-настоящему в этой сессии
   из-за отсутствия доступа к БД в sandbox. Это первоочередная задача перед тем, как считать
   интеграцию завершённой.
2. Заменить самодельный `web/src/api/client.ts` на typed-клиент (`openapi-fetch` +
   `openapi-typescript` по `01-analysis/api`) для автогенерации типов запрос/ответ вместо ручных
   интерфейсов в `web/src/api/types.ts` — снизит риск повторного рассинхрона контракта, как это
   уже один раз случилось с `SlotsService`/`CatalogService`/`ProfileService` (см. отчёт о найденном
   и исправленном баге ниже).
3. Доменные политики в `src/domain/policies.ts` — клиентское *превью*; финальные значения (цена,
   доступность, факт ранней/поздней отмены) должны приходить с сервера, что уже так после
   переключения на реальные ответы `createBooking`/`cancelBooking` — локальный расчёт остаётся
   только для мгновенной обратной связи до ответа сервера.
4. Добавить React Error Boundary и фильтр по периоду дат на SCR-04 перед выходом в прод.
5. Прогнать ручной чек-лист P0-критериев приёмки в реальном браузере против живого backend —
   статическая проверка (типы/сборка/линт) не заменяет визуальную и сетевую.
6. Рассмотреть проактивный refresh токена *до* истечения (по `expires_at` из
   `tokenStorage.ts`), а не только реактивно на 401 — снизит число лишних round-trip'ов.
